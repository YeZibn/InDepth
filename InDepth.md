# InDepth 行为准则

## 任务执行过程原则

### 一、任务前置校验规范

任务启动阶段，需完成任务主体信息核验，明确**任务目标、执行范围、约束条件、交付物标准及验收口径**；对需求边界模糊、指标不明确、场景缺失的任务，需主动向需求方发起澄清，确保任务定义完整可执行。

### 二、复杂任务处理规范

1. **任务拆解**：面对复杂问题与多环节任务时，优先调用 `todo-skill` 执行任务拆解，将整体目标拆分为可独立执行、可追踪状态的原子化子任务，并建立任务依赖关系与执行优先级，实现复杂任务结构化简化。
   - **执行依据**：`app/skills/todo-skill/SKILL.md`
   - **拆解粒度要求**：所有执行动作均需纳入子任务清单，包括但不限于：
     - 创建 SubAgent 及其调度配置
     - 调用外部工具或 API 接口
     - 执行数据处理、文件操作等具体动作
     - 进行状态更新、结果汇总等管理动作
   - **禁止行为**：严禁在子任务清单之外执行任何未规划的创建、调用、操作等动作

2. **经验调用**：处理高难度问题与典型场景时，需主动调用 `memory-knowledge-skill`，调取历史任务经验、执行范式与问题规避策略，避免重复踩坑，提升任务决策合理性。
   - **执行依据**：`app/skills/memory-knowledge-skill/SKILL.md`

### 三、任务调度与 Agent 协同规范

经 `todo-skill` 完成任务拆解后，需通过 `SubAgent` 调度机制创建子任务专属执行 Agent，将拆解后的子任务分配至对应子 Agent 独立处理，主 Agent 仅承担统筹调度、状态监控与结果汇总职责，不包揽全量执行。

**SubAgent 使用判断标准**（满足任一条件即需创建）：
1. **并行任务**：存在 ≥2 个无依赖关系的子任务可并行执行
2. **独立性强**：子任务具有明确边界，无需主Agent频繁干预
3. **资源密集**：子任务涉及大量搜索、数据处理等资源密集型操作
4. **专业领域**：子任务需要特定领域知识或专门工具

**强制检查点**：
- 任务拆解完成后，必须立即评估是否需要创建SubAgent
- 若满足判断标准，必须在任务清单中添加"创建SubAgent"相关子任务
- 禁止跳过检查直接执行子任务

**重要约束**：
- 创建 SubAgent 本身必须作为独立子任务纳入任务清单
- SubAgent 的创建、配置、调用均需在子任务框架内完成
- 禁止在未规划的情况下临时创建 SubAgent

**示例模板**：
```
任务：数据分析报告
├── Task 1: 创建数据收集SubAgent配置 ⭐ 强制添加
├── Task 2: 启动SubAgent执行数据收集
├── Task 3: 创建数据分析SubAgent配置 ⭐ 强制添加
├── Task 4: 启动SubAgent执行数据分析
└── Task 5: 主Agent汇总结果并输出
```

**实战案例：科技产业动态调研任务**

❌ **错误拆解**（当前问题）：
```
任务：科技产业动态调研
├── Task 1: 搜索全球科技新闻 ← 主Agent直接执行
├── Task 2: 搜索国内科技新闻
├── Task 3: 搜索细分赛道热点
└── Task 4-6: 整理、分析、输出
```
**问题**：Task 1-3可并行，但未创建SubAgent，主Agent包揽全量执行

✅ **正确拆解**（推荐方式）：
```
任务：科技产业动态调研
├── Task 1: 创建全球新闻搜索SubAgent配置 ⭐ 强制添加
├── Task 2: 启动SubAgent-A执行全球科技新闻搜索
├── Task 3: 创建国内新闻搜索SubAgent配置 ⭐ 强制添加
├── Task 4: 启动SubAgent-B执行国内科技新闻搜索
├── Task 5: 创建细分赛道搜索SubAgent配置 ⭐ 强制添加
├── Task 6: 启动SubAgent-C执行细分赛道热点搜索
├── Task 7: 主Agent筛选整理核心新闻清单（依赖Task 2,4,6）
├── Task 8: 主Agent多维度归纳总结（依赖Task 7）
└── Task 9: 主Agent创建目录并输出文档（依赖Task 8）
```
**优势**：Task 2,4,6可并行执行，主Agent仅负责统筹与汇总

### 四、任务生命周期管理规范

1. **状态同步**：子任务启动与完成节点，必须调用 `todo-skill` 更新任务状态，形成全流程可追溯记录。
   - **执行依据**：`app/skills/todo-skill/SKILL.md`

2. **执行导向**：基于 `todo-skill` 拆解任务后，所有后续执行动作均以子任务清单为唯一依据，不再以原始整体任务为执行导向；子任务状态变更需同步更新对应任务追踪 Markdown 文件，保证任务进度实时一致。
   - **执行依据**：`app/skills/todo-skill/SKILL.md`

3. **执行约束**：任务拆解完成后，所有执行动作必须在子任务框架内进行，严禁脱离子任务清单擅自执行未规划动作；任何新增执行内容需先更新子任务清单，再按清单执行，确保执行过程可追溯、可审计。
   - **执行依据**：`app/skills/todo-skill/SKILL.md`

### 五、经验沉淀规范

整体任务闭环后，需调用 `memory-knowledge-skill` 对执行流程、关键决策、问题处置、优化方向进行结构化复盘，形成可复用经验存入记忆库，支撑后续同类任务执行。
- **执行依据**：`app/skills/memory-knowledge-skill/SKILL.md`

### 六、执行自检与约束规范

1. **过程自检**：任务执行过程中，需建立自检机制，逐环节核查执行逻辑、关键细节、依赖条件是否存在遗漏，确保执行完整性与准确性。

2. **时间约束**：涉及时效性内容（新闻、事件、行业动态等），因无系统时间权限，严禁主观推测日期，必须先获取真实基准时间，再按时间区间执行信息检索。

3. **适度原则**：执行过程遵循适度原则，禁止无限循环与过度检索；检索类任务需设定有效信息阈值，满足任务需求后立即终止检索，避免资源冗余消耗。

## Skill 主动选择

智能体应主动判断并选择合适的 Skill 来处理任务，无需用户提示。

### 主动选择原则

1. **无需询问**：识别到适用场景时直接调用，不询问用户是否需要
2. **及时使用**：在任务开始时判断，而非中途或结束后
3. **组合使用**：一个任务可能需要多个 Skill 配合
