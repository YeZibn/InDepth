# InDepth 运行时执行协议

本协议定义 InDepth 在任务执行过程中的强制规则与推荐实践，目标是确保：
- 可执行（规则可落地）
- 可追溯（过程可审计）
- 可复用（经验可沉淀）

术语约定：
- **MUST**：强制要求，不可违反
- **SHOULD**：推荐执行，若不执行需记录理由
- **MAY**：可选策略，按场景采用

---

## 1. 任务启动协议

### 1.1 前置校验

1. Agent **MUST** 在任务开始前明确以下信息：
   - 任务目标
   - 执行范围
   - 约束条件
   - 交付标准
   - 验收口径

2. 若存在边界不清、指标不明确、场景缺失，Agent **MUST** 先发起澄清，再进入执行。

3. 对复杂任务，Agent **SHOULD** 在启动阶段同步完成：
   - 任务拆解规划（`todo_tool`）
   - 历史经验检索（`memory-knowledge-skill`）

---

## 2. 任务拆解与执行边界协议

### 2.1 拆解触发条件

当满足任一条件时，Agent **MUST** 进行任务拆解：
- 至少 3 个可识别执行步骤
- 涉及跨文件或跨组件修改
- 预计执行时长超过 10 分钟
- 存在明确依赖关系或并行机会

### 2.2 拆解粒度要求

拆解结果 **MUST** 覆盖完整执行链路，包括但不限于：
- Agent 创建与调度动作
- 工具调用/API 调用
- 数据处理与文件操作
- 状态更新与结果汇总

### 2.3 执行边界要求

1. 拆解完成后，后续动作 **MUST** 以子任务清单为唯一执行依据。
2. 未在清单中的新增动作，**MUST** 先补充到清单，再执行。
3. Agent **MUST NOT** 在清单之外执行未规划动作。

执行依据：
- `app/tool/todo_tool/todo_tool.py`

---

## 3. SubAgent 协同协议

### 3.1 角色分工

- 主 Agent：统筹调度、依赖管理、状态监控、结果汇总
- SubAgent：执行分配的子任务

主 Agent **SHOULD NOT** 包揽全部执行动作（除非满足例外条件并记录理由）。

### 3.2 SubAgent 决策规则

满足任一条件时，主 Agent **SHOULD** 创建 SubAgent：
- 存在两个及以上可并行子任务
- 子任务边界清晰、可独立推进
- 子任务属于资源密集型（大量检索/数据处理）
- 子任务需要特定领域能力或专门工具

### 3.3 不创建 SubAgent 的例外条件

满足任一条件时，主 Agent **MAY** 不创建 SubAgent：
- 任务总耗时预计低于 10 分钟
- 并行收益低于上下文切换成本
- 外部资源限制导致并行不可用

若选择不创建，主 Agent **MUST** 在任务记录中写明：
- 不创建原因
- 替代执行策略

### 3.4 强制检查点

在任务拆解完成后，主 Agent **MUST** 立即进行一次 SubAgent 评估并记录结论。

执行依据：
- `app/tool/sub_agent_tool/sub_agent_tool.py`

---

## 4. 状态管理与审计协议

### 4.1 状态更新

1. 子任务进入执行时，状态 **MUST** 从 `pending` 更新为 `in-progress`。
2. 子任务完成时，状态 **MUST** 更新为 `completed`。
3. 发现阻塞时，状态 **MUST** 按事实回退并标注阻塞原因。

### 4.2 审计要求

每次关键状态变更 **MUST** 可追溯，至少包含：
- 时间
- 变更前后状态
- 执行动作摘要
- 责任 Agent

执行依据：
- `app/tool/todo_tool/todo_tool.py`

---

## 5. 时效性信息协议

1. 涉及时效内容（新闻、事件、行业动态）时，Agent **MUST** 先确定时间基准。
2. 输出中 **MUST** 标注时间基准（例如 `UTC+8, YYYY-MM-DD HH:mm`）。
3. Agent **MUST NOT** 主观臆测“最新”或“当前时间范围”。

---

## 6. 例外处理协议（应急直行）

当出现高优先级阻塞或应急场景时，允许“先执行后补录”，但满足以下条件：

1. Agent **MAY** 先执行必要动作以解除阻塞。
2. Agent **MUST** 在本次会话内尽快补录该动作到任务清单。
3. Agent **MUST** 记录应急原因、风险评估与回补结果。

---

## 7. 经验沉淀协议

任务闭环后，Agent **SHOULD** 进行结构化复盘，并沉淀到记忆系统：
- 关键决策依据
- 成功路径与失败路径
- 风险与规避策略
- 可复用模板与执行范式

执行依据：
- `app/skills/memory-knowledge-skill/SKILL.md`

---

## 8. Skill 主动选择协议

1. Agent **MUST** 在任务开始阶段主动判断所需能力，不等待用户提示。
2. Agent **SHOULD** 组合使用多个能力模块完成复杂任务。
3. Agent **MUST** 记录关键能力调用决策（为何调用/为何不调用）。

---

## 9. 最小执行闭环（Checklist）

每个复杂任务至少完成以下闭环：
1. 前置校验完成
2. 任务拆解完成
3. SubAgent 评估完成
4. 执行与状态同步完成
5. 结果汇总与交付完成
6. 复盘沉淀完成

若任一步骤缺失，Agent **MUST** 在交付前补齐或说明原因。
