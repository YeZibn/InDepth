# S12-T6 设计文档同步机制（V1）

更新时间：2026-04-23  
状态：Draft  
对应任务：`S12-T6`

## 1. 目标

本任务用于定义 `runtime-v2` 第一版设计文档与项目总管理表的同步机制。

本任务只回答三件事：

1. 每篇设计文档的头部最少应包含什么
2. 设计文档允许使用哪些正式状态
3. 总计划表如何统一统计各子任务文档状态

## 2. 正式结论

第一版采用以下同步规则：

1. 每篇设计文档头部统一保留固定字段
2. 文档状态只允许使用 3 个正式值
3. 项目总管理使用一张统一子任务表追踪文档状态
4. 子任务表中的状态应与对应文档头状态保持一致

## 3. 文档头最小模板

第一版每篇设计文档头部至少保留：

1. 标题
2. 更新时间
3. 状态
4. 对应任务

推荐形态如下：

```md
# 文档标题

更新时间：2026-04-23  
状态：Draft  
对应任务：`Sx-Ty`
```

## 4. 正式文档状态

第一版文档状态只允许使用以下 3 个值：

1. `Draft`
2. `Implementation`
3. `Deprecated`

## 4.1 `Draft`

表示：

1. 当前仍处于讨论与收敛阶段
2. 尚未作为正式实施基线

## 4.2 `Implementation`

表示：

1. 当前已作为正式落地依据
2. 后续实现默认按该文档执行

## 4.3 `Deprecated`

表示：

1. 当前文档已被替代
2. 只保留历史追溯价值

## 5. 总管理表规则

第一版要求在总计划文档中维护统一子任务管理表。

该表至少保留以下列：

1. `task_id`
2. `structure`
3. `title`
4. `doc_status`
5. `doc_path`
6. `wave`
7. `notes`

## 6. 同步规则

第一版采用以下同步规则：

1. 新增子任务文档时，必须在总管理表中补对应记录
2. 子任务文档状态更新时，总管理表中的 `doc_status` 必须同步更新
3. 若文档进入 `Deprecated`，总管理表也必须同步标记
4. 不允许总管理表与子任务文档头状态长期不一致

## 7. 为什么只保留文档状态

第一版不额外引入 `delivery_status` 或其他执行状态，原因如下：

1. 当前最核心的是设计基线管理
2. 额外状态容易让总表过重
3. 文档状态已经足够表达当前讨论、实施、废弃三类阶段

## 8. 对总计划表的要求

总计划表应同时承担两类作用：

1. 记录整体波次与推进顺序
2. 作为统一子任务文档索引与状态总表

因此后续更新总计划表时，应优先更新：

1. 当前完成度
2. 剩余任务列表
3. 子任务管理表

## 9. 对后续任务的直接输入

`S12-T6` 直接服务：

1. 后续所有新增设计文档
2. 总计划表的统一管理方式
3. `S9 / S10` 大批量子任务推进时的状态统计

## 10. 本任务结论摘要

可以压缩成 5 句话：

1. 每篇设计文档头部固定保留标题、更新时间、状态、对应任务
2. 文档状态只允许 `Draft / Implementation / Deprecated`
3. 总计划表必须维护统一子任务管理表
4. 子任务表中的 `doc_status` 要和文档头状态一致
5. 第一版不额外引入其他交付状态
