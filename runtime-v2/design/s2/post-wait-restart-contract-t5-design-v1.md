# S2-T5 等待后重启协议（V1）

更新时间：2026-04-25  
状态：Draft  
对应任务：`S2-T5`

## 1. 目标

本任务用于定义 `runtime-v2` 第一版中，在等待用户回复、验证结果、subagent 结果或工具结果之后，宿主层如何继续推进任务。

本任务不再讨论：

1. 独立 `resume-run` 协议
2. 旧 `run_id` 的恢复续接规则
3. 任意中途崩溃态的恢复机制
4. runtime core 内部如何从消息历史猜测等待原因

这里只回答四件事：

1. 第一版是否保留独立 `resume-run`
2. 等待后继续推进时，宿主层应如何发起执行
3. 旧 run 与新 run 的关系是什么
4. 等待来源应如何进入下一次执行

## 2. 正式结论

第一版正式结论如下：

1. 第一版不再保留独立 `resume-run` 协议
2. 所有“等待后继续推进”统一退化为：
   - 基于已有上下文重新启动一个新的 `run`
3. 宿主层不负责恢复旧 `run_id` 的执行状态
4. 宿主层只负责发起一次新的 `start-run`
5. 因此第一版正式保留的只有：
   - `start-run contract`
6. 旧 run 的作用是：
   - 提供历史上下文
   - 提供等待原因
   - 提供先前产物
   - 提供待消费外部信号
7. 新一轮推进始终生成新的 `run_id`
8. 第一版不存在“继续同一个 run”这件事
9. 第一版不支持对既有 `run_id` 做正式续接
10. `awaiting_user_input / verification / subagent / tool` 都统一理解为：
   - 下一次新 run 的输入来源
11. 宿主层不再区分 `start-run` 与 `resume-run` 两条对外协议
12. `RuntimeHost.submit_user_input(...)` 继续作为唯一正式执行入口

## 3. 为什么取消独立 `resume-run`

第一版明确取消独立 `resume-run` 协议。

原因如下：

1. 宿主协议可以明显简化
2. 不需要再为恢复设计独立宿主状态机
3. 不需要定义“哪些 paused state 可以恢复、如何恢复”
4. `run_id` 语义更稳定，每次执行实例都对应一次新的 run

因此第一版选择的不是：

1. 继续同一个 run

而是：

1. 在已有上下文基础上重新开启一个新 run

## 4. 宿主层正式语义

第一版中，宿主层对“等待后继续推进”的正式语义如下：

1. 等待中的旧 run 已经结束自己的本轮执行
2. 外部信号到达后，host 发起新的 `start-run`
3. runtime core 接收的是一个新的 `run_id`
4. 新 run 通过上下文装配读取上一轮留下的等待信息与相关产物

这意味着：

1. host 不负责恢复旧执行现场
2. host 不负责把旧 `run_id` 标回 active
3. host 只负责发起新的运行实例

## 5. 旧 Run 与新 Run 的关系

第一版中，旧 run 与新 run 的关系如下：

### 旧 run

作用：

1. 作为上一轮执行实例的历史记录
2. 保存等待原因、先前结果、上下文线索和外部信号引用

### 新 run

作用：

1. 作为等待后继续推进的正式执行实例
2. 消费旧 run 留下的可用上下文与待处理信号

第一版明确规定：

1. 新 run 不复用旧 `run_id`
2. 新 run 不宣称自己是旧 run 的同一次执行延续
3. 两者是同一 `task_id` 下的两个独立执行实例

## 6. 标识模型上的含义

本任务与 `S2-T3` 对齐后的正式含义如下：

1. `session_id` 继续稳定代表宿主会话
2. `task_id` 继续稳定代表当前任务上下文
3. `run_id` 始终只代表一次具体执行实例

因此等待后继续推进时：

1. 复用 `session_id`
2. 复用 `task_id`
3. 生成新的 `run_id`

这正是取消 `resume-run` 后标识模型保持清晰的关键。

## 7. 等待来源如何进入下一次执行

第一版中，等待来源不再触发“恢复旧 run”，而是作为新 run 的输入来源进入下一次执行。

第一版至少包括以下几类等待来源：

1. 用户回复
2. verification 结果
3. subagent 结果
4. tool 结果

这些来源的统一定位是：

1. 它们都不是“恢复命令”
2. 它们都是下一次新 run 的输入来源或上下文来源

## 8. 与 `submit_user_input(...)` 的关系

第一版保持：

1. `RuntimeHost.submit_user_input(...)` 是唯一正式执行入口

这意味着：

1. 从外层看，不需要再区分“启动新 run”和“恢复旧 run”
2. host 内部只需要决定：
   - 当前是否已有 `task_id`
   - 当前是否需要生成新的 `run_id`
   - 当前要把哪些既有上下文和等待来源带入新 run

## 9. Host 前置处理规则

在等待后重启的场景中，host 前置处理至少包括：

1. 确认当前 `task_id` 仍然有效
2. 生成新的 `run_id`
3. 读取上一轮与当前等待来源相关的宿主态或上下文引用
4. 调用已有 `start-run contract`

第一版不要求：

1. host 维护独立恢复状态机
2. host 对旧 run 进行现场级恢复

## 10. Runtime 侧输入含义

第一版中，runtime core 收到的仍然是一次新的 `start-run` 输入。

也就是说：

1. runtime core 不需要再判断这是 `start` 还是 `resume`
2. runtime core 接收的是一个新的 `run_id`
3. 等待后的继续推进，靠输入上下文与外部信号来源表达，而不是靠 `is_resume=true`

因此本任务与 `S2-T4` 对齐后的方向是：

1. `StartRunInput` 应逐步去掉 `is_resume` 这种区分位
2. 第一版正式协议只保留“新 run 启动”

## 11. 第一版不做的事情

第一版中，`S2-T5` 明确不做以下设计：

1. 不定义独立 `resume-run contract`
2. 不定义旧 `run_id` 的复用规则
3. 不定义 paused run 的宿主级恢复状态机
4. 不支持对任意旧 run 直接盲恢复
5. 不支持“恢复时顺便重建同一个 run”

## 12. 与其他任务的关系

`S2-T5` 直接依赖：

1. `S2-T2` `RuntimeHost` 接口草案
2. `S2-T3` 标识生命周期模型
3. `S2-T4` 新 run 启动协议

`S2-T5` 直接服务：

1. `S12` 事件与宿主态对齐
2. 等待后继续推进的宿主实现
3. 总计划中第一版宿主协议的最终收口

## 13. 本任务结论摘要

可以压缩成 6 句话：

1. 第一版取消独立 `resume-run` 协议
2. 所有等待后继续推进统一改为基于已有上下文重开新 run
3. host 不恢复旧 `run_id`，只发起新的 `start-run`
4. 旧 run 负责留下历史上下文、等待原因和先前产物
5. 下一次推进复用 `session_id / task_id`，但必须生成新的 `run_id`
6. 第一版正式保留的宿主执行协议只有 `start-run contract`
