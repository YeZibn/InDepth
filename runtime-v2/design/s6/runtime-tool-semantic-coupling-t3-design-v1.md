# S6-T3 Runtime 与工具语义耦合策略（V1）

更新时间：2026-04-21  
状态：Draft  
对应任务：`S6-T3`

## 1. 目标

本任务用于定义 `runtime-v2` 中 runtime 与工具语义之间的耦合边界。

目标是：

1. 说明什么叫“语义感知”
2. 明确 runtime 该理解哪些工具的含义
3. 明确哪些工具只按统一协议处理
4. 避免 runtime 重新变成吞掉所有工具细节的上帝对象

## 2. 语义感知的正式定义

这里的“语义感知”不是指 runtime 去理解自然语言，而是指：

`runtime 是否把某个工具调用结果，当成会直接改变主状态流的正式信号来处理。`

也就是说，如果 runtime 对某个工具有语义感知，它会在工具结果之外进一步知道：

1. 这个工具属于哪个结构域
2. 这个工具的成功/失败会影响哪类主状态
3. 工具执行后是否需要更新 `RunContext`、`TaskGraphState` 或主链路判断

## 3. 正式结论

本任务最终结论如下：

1. runtime 只对少数主链路关键工具保留显式语义感知
2. `task graph / node` 相关工具属于主链路状态工具
3. 基础执行工具一律不做业务语义感知
4. `subagent` 工具第一版只做域级识别，不深入语义细节
5. `search` / `memory` 工具第一版只做域级识别，不进入主状态流核心

## 4. 工具分层

第一版统一把工具分成 3 类：

1. 主链路状态工具
2. 通用执行工具
3. 旁路能力工具

## 4.1 主链路状态工具

这一类工具会直接改变主状态流，因此 runtime 必须保留显式语义感知。

当前应归入这一类的工具包括：

1. `plan_task_graph`
2. `update_node_status`
3. `get_next_node`
4. `reopen_node`
5. `append_followup_nodes`

对这一类工具，runtime 不只是记录工具结果，还要知道：

1. graph 是否被创建或更新
2. active node 是否变化
3. 是否需要推进或回退执行骨架
4. 是否会影响后续 phase 或主链路判断

## 4.2 通用执行工具

这一类工具是通用能力，runtime 不应理解它们的业务语义。

当前典型工具包括：

1. `bash`
2. `read_file`
3. `write_file`
4. `get_current_time`

对这一类工具，runtime 只做：

1. 统一协议处理
2. 事件记录
3. 结果摘要收集

runtime 不应做：

1. 推断这个工具调用在业务上意味着什么
2. 根据工具名直接修改 task graph

## 4.3 旁路能力工具

这一类工具不直接进入主状态流核心，但 runtime 可以知道它们属于哪个域。

当前第一版包括：

1. `memory` 相关工具
2. `search` 相关工具
3. `subagent` 相关工具

对这一类工具，runtime 第一版只保留域级识别：

1. 它知道这是 `memory` / `search` / `subagent` 域
2. 但不深入理解具体业务语义

## 5. 三类工具的处理差异

可以用下面这张表概括：

| 工具类型 | runtime 是否语义感知 | 是否直接影响主状态流 |
|---|---|---|
| 主链路状态工具 | 是 | 是 |
| 通用执行工具 | 否 | 否 |
| 旁路能力工具 | 只做域级识别 | 否 |

## 6. 为什么不做两种极端

本任务明确避免两种极端：

### 极端 1：完全不理解任何工具语义

问题：

1. 那么 `task graph` 相关工具的结果就无法进入主状态流
2. runtime 会失去执行骨架推进能力

### 极端 2：runtime 理解所有工具的业务含义

问题：

1. runtime 会重新吞入大量领域细节
2. 很快又回到单体 `AgentRuntime`

因此第一版采用中间路线：

1. 只对主链路关键工具保留最小语义感知
2. 其余工具只按协议或域级别处理

## 7. 与 `task graph` 的关系

本任务与 `S5-T2`、`S5-T3` 已对齐：

1. v2 使用 `task graph`
2. 最小执行单元是 `node`

因此：

1. 图级工具进入主链路状态工具集合
2. 节点级工具进入主链路状态工具集合
3. runtime 对它们保留最小语义感知是合理且必要的

## 8. 与 `meta.category` 的关系

本任务与 `S6-T2` 对齐：

1. 所有工具都有统一信封
2. 所有工具都有 `meta.category`

这里进一步规定：

1. runtime 可以先依据 `meta.category` 做第一层判断
2. 然后只对白名单中的主链路状态工具做进一步语义处理

也就是说：

1. 统一协议是基础
2. 语义感知是少量增量规则

## 9. 对其他任务的直接输入

`S6-T3` 直接服务：

1. `S6-T4` 工具分域结构
2. `S6-T5` tool call 进入状态流 / 事件流 / 证据链的路径
3. `S3-T4` step loop 最小职责
4. `S5-T4` 执行图关系模型

同时它直接依赖：

1. `S5-T2` task graph 命名决策
2. `S5-T3` 最小执行单元定义
3. `S6-T2` 统一 tool protocol

## 10. 本任务结论摘要

可以压缩成 5 句话：

1. 语义感知是指 runtime 是否把工具结果当成主状态流信号处理
2. runtime 只对少数主链路关键工具保留显式语义感知
3. `task graph / node` 相关工具属于这类关键工具
4. 基础执行工具只按统一协议处理，不做业务语义感知
5. `memory` / `search` / `subagent` 工具第一版只做域级识别
