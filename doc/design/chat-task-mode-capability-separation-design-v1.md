# InDepth 单一 Task 模式统一能力设计 V1

更新时间：2026-04-16  
状态：Implemented（已完成）

当前结论（2026-04-16）：
1. 去除 `chat` 模式，CLI 只保留 `task` 单模式。
2. `task` 模式下所有工具默认可用（含搜索、todo、本地执行工具）。
3. 不引入高风险二次确认闸门。
4. 现有双模式相关实现已回收为单模式策略。

当前落地状态（2026-04-16）：
1. `runtime_agent` 已移除 chat 执行拦截与 chat 模式提示逻辑。
2. CLI 默认进入 `task` 模式，并以 `task` 新任务启动。
3. `/mode` 保留兼容提示：仅支持 `task` 单模式；`/task`、`/newtask`、`/new` 可直接开启新任务。
4. 普通输入统一走执行链路，不再按模式禁用工具。
5. 相关单测已更新并通过验收。

## 1. 背景与问题

此前双模式（`chat/task`）方案在实践中出现两个问题：
1. `chat` 模式禁用工具过重，影响效率（搜索、todo 等也被连带限制）。
2. 频繁切模式与上下文边界管理，增加交互成本和心智负担。

## 2. 目标

1. 采用单一 `task` 模式，避免模式切换成本。
2. 保持执行效率：所有工具默认可用，支持连续执行。
3. 降低交互摩擦：不引入确认往返。

## 3. 非目标

1. 不引入新的自动任务分类器。
2. 不改变 Runtime 主循环协议。
3. 不在本期引入 GUI 级权限系统。

## 4. 能力策略

`task` 模式统一能力：所有已注册工具默认可用，不做风险分级拦截。  
示例包括：
1. 搜索工具（web/url/search guard）。
2. todo 工具（创建/更新/查询任务状态）。
3. 文件读写与补丁修改。
4. 本地命令执行与测试运行。
5. 自动化与任务链路相关工具（按系统注册策略启用）。

## 5. 交互规则

1. CLI 默认进入 `task` 模式并持续在该模式运行。
2. 用户提出执行请求后，按现有执行链路直接执行，不增加确认回合。
3. 通过日志与测试保障稳定性，不通过交互闸门限制能力。

## 6. 实现建议

### 6.1 CLI 层

1. 移除 `/mode chat` 与相关提示逻辑。
2. 保留 `/task`、`/newtask`、`/new`、`/status` 等任务命令。
3. 移除高风险操作确认状态机逻辑（如存在）。

### 6.2 Agent/Runtime 层

1. 默认不再整体禁用工具。
2. 不增加工具调用前的风险级别判定与确认闸门。
3. 保留现有执行事件日志与观测链路。

### 6.3 配置层

建议新增配置项（示意）：
1. `runtime.single_mode_enabled=true`
2. `runtime.default_mode=task`
3. `runtime.task_mode_all_tools_enabled=true`

## 7. 观测与验收

### 7.1 验收标准

1. 搜索、todo 在默认路径可直接执行。
2. 文件写入、命令执行等操作不需二次确认即可执行。
3. 单模式下任务命令与执行成功率不回归。
4. 工具调用日志完整可观测。

### 7.2 观测指标

1. `tool_call_success_rate`：工具调用成功率。
2. `task_completion_rate`：任务完成率。
3. `search_todo_usage_rate`：搜索与 todo 工具使用占比。
4. `execution_latency_p95`：端到端执行延迟 p95。

## 8. 风险与缓解

风险：
1. 无确认闸门下，误执行副作用操作的代价更高。
2. 工具全开可能放大提示词误解带来的执行偏差。

缓解：
1. 强化系统提示中的执行约束与目标对齐要求。
2. 对关键工具链路补充回归测试与失败回滚指引。
3. 通过观测指标持续监控失败率与异常操作。

## 9. 里程碑建议

1. M1：移除 chat 模式入口，恢复单模式工具全可用。
2. M2：清理风险闸门相关逻辑与配置项。
3. M3：接入观测指标并做稳定性优化。
